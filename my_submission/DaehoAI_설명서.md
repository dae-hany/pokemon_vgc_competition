# 🏆 DaehoAI 기술 아키텍처 및 동작 원리 (최종 보고서)

본 문서는 2026년 CoG VGC AI Competition 출품작인 `DaehoAI`의 기술적 설계와 핵심 알고리즘을 소스 코드 기반으로 상세히 기술한 전문 설명서입니다.

## 1. 시스템 아키텍처 개요 (System Overview)

`DaehoAI`는 날씨를 인지하고 활용하는 **날씨 인식형 듀얼 플랜(Weather-aware Dual-Plan)** 아키텍처를 채택하고 있습니다. 비(Rain)와 햇살(Sun) 두 가지 날씨 전략을 동시에 보유하며, 어떤 환경에서도 최적의 성능을 발휘하도록 3개의 정책 모듈이 유기적으로 작동합니다.

```
DaehoCompetitor
├── SmartTeamBuildPolicy    → [날씨 최적화] 50마리 로스터 → 듀얼 날씨 6마리 팀 구성
├── CoverageSelectionPolicy  → [견고성 평가] 6마리 엔트리 → 전수 탐색을 통한 최적 4마리 선발
└── ChampionshipBattlePolicy → [날씨 제어] MCTS 기반 전투 수행 + 날씨 강제권Wrapper
```

## 2. Phase 1: 팀 빌딩 — `SmartTeamBuildPolicy`

> **파일**: `team_build_policy.py` | **사용 트랙**: Championship Track 전용

단순한 상성 보완을 넘어, **5가지 날씨 상태(CLEAR, RAIN, SUN, SAND, SNOW)** 전체에서 균등하게 강력한 팀을 구성합니다.

### 2-1. 5단계 날씨 데미지 매트릭스 구축

로스터 내 모든 포켓몬 쌍에 대해, 엔진의 날씨 보정치를 반영한 데미지 행렬을 구축합니다.
- **날씨 보정**: 쾌청(SUN) 시 불꽃 타입 위력 1.5배/물 타입 0.5배, 비(RAIN) 시 물 타입 1.5배/불꽃 타입 0.5배 등 엔진 로직 반영.
- **방어 보정**: 모래바람(SAND) 시 바위 타입 특방 1.5배, 눈(SNOW) 시 얼음 타입 방어 1.5배 보정 반영.

### 2-2. 견고한 목적 함수 (Robust Objective Function)

특정 날씨에만 강한 팀이 아닌, 어떤 날씨에서도 무너지지 않는 **견고성(Robustness)**을 목표로 합니다.
```python
improvement = 0.7 * min_weather_improvement + 0.3 * mean_weather_improvement
```
- **최악의 날씨(min)** 성능에 70%, **평균(mean)** 성능에 30%의 가중치를 두어, 약점이 없는 "Robust"한 커버리지를 확보합니다.

### 2-3. 듀얼 플랜 제약 조건 (Dual-Plan Shaping)

단일 날씨에 올인하는 리스크를 피하기 위해 소프트 제약 조건(Bonus/Penalty)을 적용합니다.
- **날씨 세터(Setter) 보너스**: 비(RAIN)와 햇살(SUN) 세터를 동시에 보유할 경우 큰 가중치(+8.0) 부여.
- **수혜자(Beneficiary) 캡**: 각 날씨의 이득을 보는 포켓몬(예: 비의 물 타입)을 1~2마리로 제한하여 오버클럭 방지.
- **세터 과잉 페널티**: 세터가 3마리 이상일 경우 효율 저하를 방지하기 위해 감점.

### 2-4. 그리디 커버리지 탐색

**Step 1**: `base_scores`(CLEAR 기준 화력+내구+스피드)가 가장 높은 포켓몬 선발.
**Step 2**: 견고한 목적 함수 + 듀얼 플랜 보너스 - 공유 약점 페널티가 가장 높은 포켓몬을 순차적으로 추가.

### 2-5. EV/성격 및 기술 선택

- **EV 배분**: 기술 구성을 분석하여 물리/특수/쌍두형으로 분류 후 최적화 (ADAMANT, MODEST, HASTY).
- **기술 선택 (`_select_best_moves`)**: 타입 다양성을 우선시하며, **날씨를 깔 수 있는 기술(weather_start)**이 있는 경우 우선적으로 채택하여 전략 가치를 보존합니다.

## 3. Phase 2: 선발 — `CoverageSelectionPolicy`

> **파일**: `selection_policy.py` | **사용 트랙**: Championship Track 전용

상대 엔트리 6마리에 맞서, 아군 6마리 중 최적의 시너지를 내는 **4마리 조합**을 수학적으로 결정합니다.

### 3-1. 전수 탐색 (Exhaustive Search)

기존의 그리디 방식 대신, 아군 6마리 중 4마리를 뽑는 **모든 조합(15가지)**을 전수 조사합니다. 조합 수가 적기 때문에(C(6,4)=15) 매 턴 정밀한 시뮬레이션이 가능합니다.

### 3-2. 날씨별 견고성 점수 (`_robust_coverage_score`)

각 조합에 대해 5가지 날씨 상태에서의 성능을 평가합니다.
1. 각 날씨(CLEAR, RAIN, SUN, SAND, SNOW)에서 상대 6마리에 대한 아군 4마리의 베스트 대응력을 계산합니다.
2. 상대방 중 가장 대응하기 힘든 **하위 25%(Weak-tail)**의 평균 데미지 비율을 산출합니다.
3. 최종 점수 = `0.7 * (최악 날씨 점수) + 0.3 * (평균 날씨 점수)`

### 3-3. 듀얼 플랜 보너스 (`_dual_plan_bonus`)

선발된 4마리가 실제 배틀에서 날씨 전략을 수행할 수 있는지 평가합니다.
- **연계성**: 비 세터와 비 수혜자, 혹은 햇살 세터와 햇살 수혜자가 조화롭게 포함되었는지 확인합니다.
- **유연성**: 비와 햇살 전략을 모두 수행 가능한 4마리 조합에 가산점을 부여하여, 배틀 중 유연한 전략 수정을 가능케 합니다.
- **오버투자 방지**: 세터나 수혜자가 너무 많아 화력 밸런스가 깨지는 조합에는 페널티를 부여합니다.

### 3-4. 내구 보정 (Bulk Bonus)

소위 '유리대포'들로만 팀이 구성되는 것을 방지하기 위해, 팀 전체의 종합 내구도(HP/DEF/SpD 정규화 합산)를 8% 확률로 가산하여 안정성을 높였습니다.

### 3-5. 최종 결정

```python
score = 1.6 * coverage_score + plan_bonus + bulk_bonus
```
위 공식에 의해 가장 높은 점수를 얻은 4마리 조합이 최종 선발됩니다.

## 4. Phase 3: 배틀 — `ChampionshipBattlePolicy`

> **파일**: `battle_policy.py` | **사용 트랙**: Battle Track + Championship Track 모두 사용

매 턴 주어지는 **100ms(실제 제한 90ms 내 실행)** 안에서 최적의 행동을 결정합니다. 특히 챔피언십 트랙에서는 날씨를 능동적으로 제어하는 로직이 추가되었습니다.

### 4-1. 날씨 제어 래퍼 (Championship Weather Wrapper)

MCTS 탐색 전, 현재 상황이 아군의 날씨 전략을 강제하기에 충분히 유리한지 판단합니다.
- **날씨 이득 점수**: `(아군의 이득 - 상대의 이득) + 교체 보너스`
- **강제 실행**: 이득 점수가 임계값(80.0)을 넘고 아군 포켓몬이 세터 기술을 보유했다면, MCTS 탐색을 생략하고 즉시 날씨를 설치합니다. 이를 통해 날씨 시너지를 극대화합니다.

### 4-2. MCTS 기반 행동 결정 (`EnhancedBattlePolicy`)

날씨 강제 상황이 아닐 경우, 고도화된 Monte Carlo Tree Search(MCTS)를 통해 수읽기를 수행합니다.

**MCTS 주요 파라미터:**

| 파라미터 | 기본값 | 의미 |
|----------|--------|------|
| `time_limit_ms` | 90ms | 실제 탐색 제한 시간 |
| `rollout_depth` | 4턴 | 롤아웃 시뮬레이션 깊이 |
| `C` (UCB 탐험 계수) | 1.41 | MCTS 탐험-활용 균형 계수 |

### 4-3. 불완전 정보 극복: Mock State 생성 (`_deduce_moves`)

엔진에서 상대방의 **미공개 기술은 `DUMMY_MOVE`**로 표시되어 시뮬레이션 중 크래시를 유발합니다. DaehoAI는 이를 다음과 같이 처리합니다:

```python
# 상대 포켓몬의 기술 슬롯 수가 max_moves(4)보다 적으면,
# 해당 포켓몬의 종족값(species)에 등록된 기술 풀에서 랜덤으로 기술을 추론·추가
ids = [m.constants.id for m in pokemon.battling_moves]  # 이미 알고 있는 기술 제외
moves = [m for m in pokemon.constants.species.moves if m.id not in ids]
pokemon.battling_moves += [BattlingMove(m) for m in random.sample(moves, ...)]
```

> 이미 공개된 기술은 제외하고, **해당 포켓몬이 배울 수 있는 기술 중에서만 랜덤 추론**함으로써 완전 무작위보다 훨씬 현실적인 시뮬레이션을 보장합니다.

### 4-3. 행동 공간 생성 (`get_all_possible_actions`)

아군 활성 포켓몬 2마리의 가능한 모든 행동 조합을 생성합니다:

```
[포켓몬1 행동 (기술 인덱스, 타겟 인덱스)] × [포켓몬2 행동 (기술 인덱스, 타겟 인덱스)]
```

각 포켓몬의 행동은 **PP > 0이고 비활성화되지 않은 기술**에 한해 생성됩니다. 특히, **스위치(Switch) 행동은 다음 조건 중 하나라도 충족될 때만 후보에 포함**하여 탐색 효율을 극대화합니다:

1.  **즉사 위험 (OHKO Threat)**: 상대방의 공격으로 이번 턴에 KO될 가능성이 있는 경우.
2.  **불리한 매치업 (Bad Matchup)**: 아군이 주는 데미지는 미미하고(15% 미만), 상대가 주는 데미지는 높은(35% 이상) 상황.
3.  **상태 이상 리스크 (Status Risk)**: 물리 공격수가 화상(Burn)에 걸리는 등 전투 효율이 급감한 경우.

> **필터링의 효과**: 무분별한 교체 탐색을 억제함으로써, 100ms라는 짧은 시간 내에 **공격 조합에 대한 시뮬레이션 깊이**를 획기적으로 높였습니다. (Yamabuki 상대 승률 34% → 54% 향상 확인)

### 4-4. 상태 평가 함수 (`evaluate_state`)

MCTS의 모든 평가는 이 함수의 반환값인 `own_score - enemy_score`로 이루어집니다.

**아군/적군 각각의 점수 계산 요소:**

| 요소 | 점수 | 의미 |
|------|------|------|
| 활성 포켓몬 HP 비율 합계 | `× 50` | HP 보존율 |
| KO된 적군 포켓몬 수 | `(4 - enemy_count) × 400` | KO 가중치 (가장 큰 점수) |
| 아군 활성 포켓몬 수 | `× 300` | 수적 우위 유지 |
| 아군 예비 포켓몬 수 | `× 100` | 후속 전력 보유 |
| 상대방 상태이상 | `+ 1000` (아군 점수) | 화상/마비 등 상태이상 유발 가치 |
| 스피드 스탯 | `+ speed` | 선공 우위 평가 |
| 스탯 변화(Boost) 합계 | `× 20` | 랭크업 가치 평가 |

**게임 종반(총 생존 수 ≤ 2) 보정**: 점수를 `× 0.8`로 줄여 종반의 극단적 평가를 완화.

> **핵심 설계 철학**: KO 1개의 가치(400점)는 HP 회복 전체(최대 50점)보다 **약 8배** 높게 설정됩니다. 이를 통해 DaehoAI는 본능적으로 **수적 우위(KO 우선)**를 추구하는 전략을 학습합니다.

### 4-5. MCTS 핵심 루프

**트리 정책 (`tree_policy`)**: 루트에서 리프까지 내려가며 탐색할 노드를 선택합니다.
- 아직 시도하지 않은 행동이 있으면 → **확장(Expand)**
- 모든 행동을 시도했으면 → **UCB1 점수 최고 자식 노드 선택**

**확장 (`expand_one_child`)**: 미시도 행동 중에서 다음 규칙으로 행동을 선택합니다:
- **80% 확률**: `GreedyBattlePolicy`가 추천하는 최선 행동 선택 (활용)
- **20% 확률**: 랜덤 행동 선택 (탐험)

상대방 행동도 동일하게 `GreedyBattlePolicy`를 통해 추론 (`opp_policy`).

**롤아웃 (`rollout`)**: 리프 노드에서 `rollout_depth`(4턴)만큼 시뮬레이션:
- 현재 상태가 **불리(score < 0)**하면: **60% 랜덤, 40% Greedy** (탈출 시도)
- 현재 상태가 **유리(score ≥ 0)**하면: **20% 랜덤, 80% Greedy** (우세 유지)

**역전파 (Backpropagation)**: 롤아웃 결과 점수를 루트까지 모든 조상 노드에 누적 (`total_reward`, `visit_count`).

**UCB1 공식** (자식 노드 선택):
```
UCB1 = Q + C × √(ln(N_parent) / N)
     = (총 보상 / 방문 횟수) + 탐험 계수 × √(부모 방문 로그 / 자신 방문 횟수)
```

### 4-6. 위기 탈출 메커니즘 (Adaptive Parameters)

MCTS 루프 중 롤아웃 결과가 `-600` 이하의 극단적 불리 상태로 나오면:
```python
if reward < -600:
    self.C = 10.0          # 탐험 계수를 대폭 높여 새로운 행동 탐색 강화
    self.rollout_depth *= 2  # 롤아웃 깊이를 2배 늘려 장기적 결과 고려
else:
    self.C = 1.41          # 정상 상태 복귀
    self.rollout_depth = 4
```

> **효과**: 위기 상황에서는 탐험성을 높이고 더 깊이 시뮬레이션하여 반전 가능성을 적극적으로 탐색합니다.

### 4-7. 최종 행동 결정 (`decision`)

- **MCTS 결과 있음**: 루트의 자식 노드 중 **UCB1 점수가 가장 높은 노드의 행동** 반환
- **MCTS 결과 없음** (탐색 실패 시): `GreedyBattlePolicy` 결과를 Fallback으로 사용

## 5. 트랙별 동작 흐름 요약

### ⚔️ Battle Track

Battle Track은 팀과 선발이 엔진에 의해 고정 제공되므로, **`EnhancedBattlePolicy`만 실행**됩니다.

```
[엔진이 팀 제공] → 매 턴: MCTS(90ms) → 최선 행동 반환
```

각 턴의 상세 흐름:
1. `decision(state)` 호출
2. `MCTS(state, 90ms)` 시작 → 90ms × 0.9 = **81ms 이내** 반복
3. `tree_policy` → `expand` 또는 `UCB1 select` → `rollout(4턴)` → `backpropagate`
4. 시간 초과 시 루트 자식 중 최고 UCB1 노드의 행동 반환
5. 탐색 결과가 없으면 `GreedyBattlePolicy` 폴백

### 🏆 Championship Track

Championship Track은 3단계 파이프라인이 **순서대로 모두 실행**됩니다.

```
[로스터 50마리 공개]
        ↓
① SmartTeamBuildPolicy: 5개 날씨 시뮬레이션 → 듀얼 날씨(비/햇살) 최적 6마리 구성
        ↓
[상대방 6마리 엔트리 공개]
        ↓
② CoverageSelectionPolicy: 전수 탐색(C6,4) → 날씨 견고성 평가 → 최적 4마리 선발
        ↓
[배틀 시작]
        ↓
③ ChampionshipBattlePolicy: 날씨 강제권 판단 → MCTS(90ms) 탐색 → 최선 행동 반환
```

## 6. 종합 요약 (Conclusion)

**`DaehoAI`**는 날씨라는 환경 변수를 전략적으로 통제하는 **지능형 기상 제어 AI**입니다.

| 단계 | 모듈 | 핵심 기술 | 목표 |
|------|------|-----------|------|
| 팀 빌딩 | `SmartTeamBuildPolicy` | 5개 날씨 매트릭스 + 듀얼 플랜 쉐이핑 | 어떤 날씨에서도 약점이 없는 듀얼 날씨 팀 구축 |
| 선발 | `CoverageSelectionPolicy` | 전수 탐색 + 날씨별 견고성(Weak-tail) 평가 | 상대 6마리를 상대로 날씨 우위를 점할 4마리 선발 |
| 배틀 | `ChampionshipBattlePolicy` | 날씨 제어 Wrapper + 고도화 MCTS | 기상 주도권 확보 및 시뮬레이션 기반 최적 수 결정 |
